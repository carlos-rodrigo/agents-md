# Simple Tasks / Work Orders

Use markdown files for execution state when work benefits from explicit delegation, sequencing, or resumption.

Primary feature-flow location:

```text
docs/features/{feature}/work-orders/
```

Legacy location:

```text
.features/{feature}/tasks/
```

Do **not** force task files for tiny or one-shot changes. Small approved features can execute directly from strategy/model/decisions/proof.

---

## Feature-flow packet shape

```text
docs/features/{feature}/
├── feature.json
├── strategy.md
├── system-model.md
├── decisions.md
├── proof.md
├── work-orders/
│   ├── README.md
│   ├── 001-do-one-thing.md
│   └── 002-do-next-thing.md
├── execution/
│   ├── README.md
│   └── 001-wo-001.md
├── diagrams/
└── review.md
```

Markdown/JSON are source of truth. `index.html` is generated by `/feature view`.

## Work Order v2 format

```markdown
---
id: WO-001
status: draft # draft | ready | blocked | done
order: 1
created: YYYY-MM-DD
---

# WO-001: {Title}

## Mission

One concrete execution mission.

## Strategic context

Relevant excerpt from strategy/system model.

## Decisions to preserve

- D-001 — ...

## Agent-owned choices

- Implementation details the agent may decide.

## Escalation triggers

- Product/system ambiguity
- Conflict with decisions
- Proof cannot be produced
- Scope expansion

## Proof required

- [ ] Targeted check
- [ ] Regression gate

## Execution report

After implementation, write an execution report under `../execution/` and mark this work order `done`.
```

### Status semantics

- `draft` — not approved for execution.
- `ready` — user approved; agent may execute.
- `blocked` — waiting on decision/dependency/proof.
- `done` — implemented and execution report exists.

Only `ready` work orders should be executed.

---

## Execution reports

Reports live in:

```text
docs/features/{feature}/execution/
```

They should include frontmatter:

```yaml
---
id: ER-001
workOrder: WO-001
status: draft # draft | complete
created: YYYY-MM-DD
---
```

Reports capture repo-relative files changed, proof results, deviations, and strategic follow-up.

## Operations

### List work orders

```bash
ls -1 docs/features/{feature}/work-orders/*.md 2>/dev/null | grep -v README
```

### Find ready work orders

```bash
grep -l "status: ready" docs/features/{feature}/work-orders/*.md
```

### Create a work order

Prefer the extension command when available:

```text
/feature work-order "{title}" --slug {feature}
```

Otherwise create the next numbered markdown file manually.

### Create an execution report

Prefer:

```text
/feature report WO-001 --slug {feature}
```

### Refresh dashboard

```text
/feature view {feature}
```

## Legacy `.features/` tasks

Use legacy tasks only when requested or when an existing repo workflow depends on them.

Legacy task location:

```text
.features/{feature}/tasks/NNN-task-name.md
```

Legacy task statuses:

```yaml
status: open # open | in-progress | done | blocked
```

A legacy task should still be self-contained: context, what to do, codebase research, patterns, acceptance criteria, files, and verify steps.

---

## Best Practices

1. **Research before splitting** — capture prior art, likely files, and verification before writing execution units.
2. **One unit = one behavior** — keep work orders/tasks small enough to complete cleanly.
3. **Self-contained execution** — do not offload missing context to future exploration.
4. **Prefer one good work order over many vague ones.**
5. **Status must be trustworthy** — never mark ready/done prematurely.
6. **Proof is part of done** — every work order needs proof and an execution report.
